# DesignDoc の執筆規範

機能・コンポーネントの設計文書（DesignDoc.md）を書くときは、以下に従う。
文章そのものの規範は japanese-tech-writing skill に定めがあるので、**必ず併せて読み込んでから書く**。

## 置き場所と言語

- 対象コンポーネントのソースと同じディレクトリに `DesignDoc.md` として置く。
- 本文は日本語。コード・識別子・固有名は原綴りのまま。

## README.md との棲み分け

- **README.md**：日々の操作リファレンス（ビルド手順、環境変数、エンドポイント一覧など）。具体値の単一の情報源。実装が変わるたび更新される。
- **DesignDoc.md**：設計判断の記録（目的、境界、構成、制約、棄却した代替案、未解決の課題）。判断が変わったときだけ更新される。
- 具体値（ポート、env 名、パス）は README にだけ書き、DesignDoc からは参照する。重複させない。
- 両ファイルの冒頭で相互に参照し、この分担を一文で宣言する。

## 節構成

次の順で構成する。これは**メニュー**であり、対象に正直に当てはまらない節（Timeline、Privacy、Legal 等）は省いてよい。逆に、埋められない節を形だけ埋めない。

1. **タイトルとメタデータ**：冒頭に表で「著者 / 作成日 / 状態 / 正本（このファイルのパス）」。著者は依頼者に確認する（既定は git の user 名）。状態は「草案 / 実装済」のような安定した値だけにする。「PR レビュー中」のようなライフサイクル依存の値は、マージした瞬間に陳腐化して更新を強いるので書かない。
2. **目的（Objective）**：この機能が何のためにあるかを一文で。実装語彙を使わない。
3. **背景**：問題の文脈。なぜ今これが必要か、既存手段はなぜ不足か。因果には機構（なぜそうなるか）を一文添える。
4. **関連文書**：README、関連 PR/issue、前提となる変更へのリンク。
5. **ゴール**：ユーザーや製品への影響で書く。実装内容で書かない（「〜コンポーネントを追加する」ではなく「検証者が〜できる」）。
6. **非ゴール**：やらないことを明示し、それぞれに理由を一文添える。スコープ論争の予防が目的。
7. **シナリオ**：完成したシステムを使う場面を、行為者を主語にした時系列で 1〜2 個。
8. **システム構成（図）**：後述の図の規範に従う。図の直後に、構成の要点を 2〜4 個の太字見出し付き段落で述べる。
9. **用語集**：文書内で定義した術語だけを載せる。一般語は載せない。
10. **制約**：toolchain・環境・依存の制限と、それが設計に与えた影響。
11. **性能目標と監視**：測定可能な目標と、その逸脱をどう検知するか。debug ツール等では軽量でよいが「実測済みか未測定か」を区別して書く。
12. **インターフェース**：設計として決めた形（API の様式、エラーの返し方）だけを書き、一覧は README へ参照。
13. **セキュリティ**：脅威モデル（誰が攻撃者か）を先に一文で定め、境界と対策を対応づける。「対策なし」の場合はその根拠を書く。
14. **ログ**：何を、なぜ記録するか。
15. **未解決の課題（Open Issues）**：問題 + 現状 + 次の一歩。設計承認をブロックしない事項の置き場。
16. **解決済みの課題（Resolved Issues）**：設計途中で遭遇し解決した論点。原因と解決を一段落で。
17. **検討した代替案**：棄却した案それぞれに棄却理由を付ける。理由は「なぜ採用案より劣るか」の機構まで書く。

## 取捨選択の基準

- 記載する決定は「**間違えたときのやり直しコストが高いもの**」に絞る（decision threshold）。自明な選択やいつでも変えられる値は書かない。
- 新任のチームメンバーが、口頭の補足なしに読み切れることを基準にする。

## 図の規範

- 図は **mermaid をコードブロックで md に直接埋め込む**（GitHub がネイティブレンダリングするため、SVG 化は不要）。
- mermaid で表現しきれない図（自由レイアウトの物理構成図など）に限り drawio で描き、SVG に export して md に埋め込む。**この場合 .drawio ソースも commit する**（編集可能性の維持）。
- 図は最小限（構成 1 枚 + 論点のある機構 1 枚程度）。図で伝わることを本文で繰り返さない。

## 誠実さ

- 実測・実機で確認した事実と、未確認の想定を区別して書く（「実測 2 秒」「長期観測していない」）。
- 検証していないことを検証済みのように書かない。性能値・挙動は測ってから書くか、未測定と明記する。

## 書き終えたら

- japanese-tech-writing の点検リスト（LLM っぽい表現、論証の厳密さ、冗長）で自己点検する。
- README との重複（具体値の二重記載）がないか確認する。
- mermaid がレンダリング可能か確認する（GitHub の preview か mermaid.live 相当の構文チェック）。
